# Summary

**Phase 7 — target topology correction**, plus the first real VPS bring-up of
staging and Tsugi itself. The P1–P6 deploy scaffold and wiki were authored
against the wrong target — the legacy `../LazyScan-Stack` (detached `aegis`
edge, `minio`, `s3.<domain>` hosts, `docker-compose.prod.yml`). The live target
is the trimmed monorepo `../LazyScan`: one nginx `web` edge (SPA + `/api`),
internal `api`/`image-svc`/`mail-svc`, **R2** storage (no minio), a single
`docker-compose.yml`, and Postgres on the box's **shared** instance. Corrected
all of it, then stood up staging and deployed Tsugi.

- **Cutover:** LazyScan `main` was an empty stub; live prod ran `development`.
  Fast-forwarded `development`→`main`, tagged + GitHub-released `v1.0.0`, repointed

  shells to `deploy.sh` and needs host git/docker/checkouts; the container is
  build+migrate only). Backfilled `v1.0.0` at `production`.

# Decisions / Constraints (logged)

- **Shared-postgres, per-env database.** prod=`lazyscan`, staging=`lazyscan_staging`
  on the one shared instance; `DATABASE_URL` is env-driven. Redis stays per-stack
  internal. Only R2 is external/shared — staging must use its own bucket.
- **Prod box compose is hand-managed (skip-worktree).** `git pull` won't touch
  `/opt/lazyscan/docker-compose.yml`, so committing the repo compose is for honesty
  only; prod is unaffected by pulls. `deploy.sh --env prod` would still need this
  reconciled if it ever drives prod from the repo.
- **Two tunnel zones:** `lazyscan.my.id` (app) ≠ `sanctuary.my.id` (infra). New app
  hostnames need a manual DNS record in the app zone; `cloudflared route dns` alone
  doesn't cover it.
- **Ports on the box:** 8080 dozzle, 8081 prod, 8082 staging.
