# Summary

Bootstrapped Tsugi from an empty repo and scaffolded **Phase 1 — Environment
Separation** (`PLAN.md`). Added project memory (`docs/wiki/`, following the
`../LazyScan` docs convention), `AGENTS.md`, and a generic `deploy/` layer with
LazyScan as the first target. Infra/config only — no Go, no live VPS.

# Files Touched

- `AGENTS.md`, `README.md`, `.gitignore`
- `docs/wiki/`: README, architecture, infra-plan (PLAN.md copy + P1 status),
  known-constraints, running, findings, resolutions, `sessions/19-06-2026.md`
- `deploy/README.md`, `deploy/bin/deploy.sh`
- `deploy/targets/lazyscan/`: `target.env`, prod+staging compose overrides,
  `.env.{prod,staging}.example`, `cloudflared/config.yml.example`

# Previous Behavior

Empty repo (LICENSE only). Production ran `dev` — every push was a potential
production deploy.

# New Behavior

`deploy/bin/deploy.sh --target lazyscan --env <prod|staging>` deploys prod from
`main` and staging from `dev` as two isolated compose stacks (project name +
port offset + env-file + branch checkout), routed by a Cloudflare Tunnel with
two hostnames. Production no longer runs `dev`.

# Reason For Change

Phase 1 objective: prevent untested code from reaching production via
environment separation on the single VPS.

# Risks

- Two fully isolated stacks roughly double RAM; the prod compose is trimmed for
  ~4 GB, so two copies will not fit a 4 GB box as-is (logged in
  known-constraints.md). A shared-backing-services variant is possible later.
- `!override` needs Docker Compose >= 2.24.4.
- `deploy.sh` is interim; the Go `release` CLI (Phase 6) replaces it.

# Notes

Verified locally: `bash -n` clean; bad env rejected (exit 2); `docker compose
config` merges clean — prod ports 8080/9000/9001, staging 8081/9100/9101, all
bound to 127.0.0.1, `!override` replaced (no double-bind). cloudflared not
installed locally — ingress YAML hand-checked. No Go code; `go.mod`/`cmd/`/
`internal/` deferred to P2.

# Summary

Scaffolded **Phase 2 — Version Visibility** (`PLAN.md`). First Go code: a stdlib
`net/http` service exposing `GET /version` → `{version, commit, deployed_at}`
plus `GET /healthz`. Module `github.com/latoulicious/Tsugi`, go 1.26, no deps.

# Files Touched

- `go.mod`, `Makefile`, `Dockerfile`
- `cmd/tsugi/main.go`, `internal/version/version.go`,
  `internal/config/config.go`, `internal/server/server.go`
- `docs/module/` (README + version/config/server) — package docs in lieu of godoc
- `docs/wiki/`: architecture (Packages + P2 status), infra-plan (P2 block),
  running (Tsugi service section), this session entry

# Previous Behavior

No Go code; repo carried only the Phase 1 deploy layer and project memory.

# New Behavior

`make build && ./bin/tsugi` serves `GET /version` and `GET /healthz` on `:8080`
(`TSUGI_ADDR`). Build identity stamped at link time (`-X` ldflags) from git via
the Makefile; the Dockerfile takes the same as `--build-arg`. `deployed_at` =
build time. Local `go build` falls back to the embedded VCS stamp.

# Reason For Change

Phase 2 objective: know exactly what is running — every deployment identifiable.

# Risks

- Tsugi's own compose / VPS deploy wiring not built yet (P5/P6); validated
  locally only (build, vet, curl).
- With no git tags, `git describe --tags --always` makes version == short SHA
  until the first tag.

# Notes

Verified: `go vet` clean, `make build` ok, `/version` + `/healthz` return
expected JSON, `POST /version` → 405 (Go 1.22+ method routing). Skills loaded:
golang-conventions, clean-code, domain-driven-design. Project preference noted:
no in-code godoc — package docs go to `docs/module/`. DDD layering deferred to
P3+ (no domain model yet).

CodeRabbit: 4 findings → fixed F-003 (WriteTimeout) + F-005 (pre-marshal
