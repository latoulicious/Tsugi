- `docs/module/{cli,git,deploy}.md` (new) + postgres/release/deployment/README;

# Previous Behavior

`release`/`deployment`/`postgres` existed but nothing drove them — no CLI, no
DB wiring, no git exec, no transactions. `deploy.sh` deployed by branch only.

# New Behavior

`tsugi release create vX` snapshots the validated staging commit + changelog and
records the release at Staging. `promote` runs the real prod deploy at the
release commit, then atomically advances Staging→Production, archives the prior
production release, and marks the deployment succeeded (failed deploy → record
`failed`, release stays staging). `rollback` redeploys an archived release via
`deploy.sh --ref <sha>` (new `Archived→Production` edge). `tsugi migrate up`
applies embedded migrations. `serve` unchanged — still starts without a DB.

# Reason For Change

Phase 6 objective: promote validated releases safely and revert production
without manual git operations.

# Risks

- Full orchestration touches the live VPS: `promote`/`rollback` run real
  `docker compose`. Validated locally by build/vet/unit tests + deploy.sh
  `bash -n`; the pgx adapter, `WithTx`, and the migration runner need a live
  Postgres (no integration test this phase).
- Rollback does a detached checkout of an older commit; still `main` lineage, so
  prod never points at `dev`. Ref charset-guarded in deploy.sh.
- Migration runner is minimal (no dirty-state recovery / down-to-version).
- `create`-deploys-to-Staging interpretation chosen (validated commit is already
  on staging per the PLAN workflow); flagged to user, no objection.

# Notes
