# Summary

Closed the last open P5.2 exit clause ("runs as host systemd"). The Go side of
P5.2 already shipped at `08d303a` (read RPCs over loopback gRPC, verified); the
only missing artifact was a host service unit. Added an `.example` systemd unit
plus doc pointers. No code changes.

# Files Touched

- `deploy/tsugi.service.example` — new. Host unit for `tsugi serve` (HTTP :8090 +
  loopback gRPC agent :8091). `EnvironmentFile=/etc/tsugi/tsugi.env`,
  `User/Group=tsugi`, `WorkingDirectory=/opt/tsugi`, `Restart=on-failure`.
- `docs/wiki/running.md` — host-service note under the write-plane agent section.
- `deploy/README.md` — unit listed in the layout tree.

# Previous Behavior

`tsugi serve` ran the agent on loopback but the repo shipped no service unit; the
runbook only mentioned the host binary via `make install`.

# New Behavior

Operators have a template unit closing P5.2's exit bar. `serve` runs as host
systemd; the agent stays loopback `:8091`, never tunneled.

# Reason For Change

P5.2 exit: "grpcurl localhost ListReleases returns rows; runs as host systemd,
unreachable via tunnel." First half was met at `08d303a`; this lands the second.

# Risks

- `User=tsugi` (not `DynamicUser=`) on purpose: P5.4's deploy RPCs will shell
  `docker`/`git`, so the account must persist and join the docker group later.
- `WorkingDirectory=/opt/tsugi` assumes `deploy/` lives there (default
  `TSUGI_DEPLOY_DIR` is relative); set an absolute path otherwise.
- Template only — actual `systemctl enable` is a VPS action (like the cloudflared
  ingress), out of repo.

# Notes

No Go touched; `go build`/`test ./...` unaffected (still green). The unit is the
only in-repo artifact P5.2 needed; the rest was code-complete.
