# Summary

Vendored the Tsugi agent gRPC contract and wired Go codegen (Yagura P5.1). The
`.proto` is authored in Yagura (`proto/tsugi_agent.proto`, source of truth); Tsugi
vendors a copy and generates the server stubs. All RPCs return Unimplemented — no
behavior, the server isn't registered yet. Read RPCs land in P5.2.

# Files Touched

- `proto/tsugi_agent.proto` — new. Vendored copy of Yagura's contract
  (`package tsugi.agent.v1`, `service TsugiAgent`).
- `internal/agentpb/tsugi_agent.pb.go`, `..._grpc.pb.go` — new, generated
  (protoc-gen-go + -go-grpc), committed so the Docker build stays protoc-free.
- `internal/agent/server.go` — new. `Server` embeds
  `agentpb.UnimplementedTsugiAgentServer`; `var _ agentpb.TsugiAgentServer`.
- `Makefile` — `proto:` target (regenerate from the vendored .proto).
- `go.mod` / `go.sum` — added `google.golang.org/grpc` + `.../protobuf`.

# Previous Behavior

No gRPC; Tsugi was HTTP `/version` + `/healthz` + the `release` CLI only.

# New Behavior

None at runtime. The package compiles and exposes a server type whose RPCs return
`codes.Unimplemented`. Not registered in `cmd/tsugi`.

# Reason For Change

P5.1 contract slice — establish the typed cross-language seam before either side
implements it (read-before-write, Yagura PLAN §11/§13).

# Risks

- Proto drift: the contract is edited in Yagura, re-copied here. Keep them in sync
  (`make proto` after copying). One file, no buf/submodule yet.
- New build-host deps for regen: protoc + protoc-gen-go + protoc-gen-go-grpc. Not
  needed for `go build` (generated code is committed).

# Notes

Generated code committed deliberately so the existing protoc-free Dockerfile keeps
working. `go build ./...`, `go vet ./...`, `go test ./...` all green.

# Summary

P5.2 — Tsugi write-plane agent skeleton (Yagura PLAN §11/§13). `tsugi serve` now
registers and serves the gRPC agent on a loopback port alongside HTTP, with the
two read RPCs implemented. Deploy/Rollback/Promote stay `Unimplemented` (P5.4).

# Files Touched

- `internal/agent/server.go` — `ListReleases`/`ListDeployments` implemented over
  two narrow consumer interfaces (`releaseLister`/`deploymentLister`) + `service`
  name. In-Go release⨝deployment join; reuses the existing repos (no new SQL).
- `internal/agent/server_test.go` — new. Fake listers assert the mapping (env
  from status, commit join, `deployed_at` fallback). stdlib only.
- `internal/config/config.go` — `AgentAddr` (`TSUGI_AGENT_ADDR`, default
  `127.0.0.1:8091`); `requireLoopback` rejects any non-loopback agent bind.
- `cmd/tsugi/main.go` — `run()` requires `TSUGI_DATABASE_URL`, opens the pool,
  registers the agent on a loopback `grpc.Server` (+reflection), serves it in a
  goroutine with HTTP, `GracefulStop` on shutdown.
- Docs: `module/{agent,config}.md`, `wiki/{running,known-constraints,infra-plan}.md`.

# Previous Behavior

`serve` ran HTTP `/version`+`/healthz` only, no DB needed. The agent package
compiled but returned `Unimplemented` and was never registered.

# New Behavior

`serve` hosts the agent on `127.0.0.1:8091` (loopback only). `grpcurl ... 
ListReleases`/`ListDeployments` return rows joined from `releases`/`deployments`;
deploy RPCs still `Unimplemented`. `serve` now **requires** `TSUGI_DATABASE_URL`
and fails fast without it.

# Reason For Change

P5.2 read-plane slice: Yagura's read plane needs a live localhost endpoint to
observe release/deployment state before the deploy RPCs land.

# Risks

- **Breaking:** `serve` now needs `TSUGI_DATABASE_URL`; the systemd unit/env must
  set it. Documented in `running.md`/`known-constraints.md`.
- Agent must stay loopback — config enforces it, but the systemd unit must not
  add a tunnel ingress for `:8091`.
- `deployed_at` for a release uses its latest deployment row, else `created_at`
  (staging releases have no deployment row) — a denormalization, not a raw column.

# Notes

Verified end-to-end against a throwaway Postgres: migrate up, seed 2 releases + 1
deployment, `tsugi serve`, `grpcurl ListReleases` → 2 rows (denormalized),
`ListDeployments` → 1 row (commit joined), `Deploy` → `Unimplemented`. Both
listeners bound `127.0.0.1` only (verified via `lsof`); a `0.0.0.0` agent bind is
rejected at config load. `go build`/`vet`/`test ./...` green.

CodeRabbit review → 5 findings (F-009–F-013). Fixed F-009 (bounded gRPC
shutdown, R-005) and F-011 (loopback IPs only, R-006). F-010 won't-fix (HTTP
plane is tunnel-fronted by design), F-012 deferred (net.Listen fail-fasts),
F-013 accepted (read-skew self-heals; FK prevents orphans).
