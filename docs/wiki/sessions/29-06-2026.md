# Summary

Vendored the Tsugi agent gRPC contract and wired Go codegen (Yagura P5.1). The
`.proto` is authored in Yagura (`proto/tsugi_agent.proto`, source of truth); Tsugi
vendors a copy and generates the server stubs. All RPCs return Unimplemented — no
behavior, the server isn't registered yet. Read RPCs land in P5.2.

# Files Touched

- `proto/tsugi_agent.proto` — new. Vendored copy of Yagura's contract
  (`package tsugi.agent.v1`, `service TsugiAgent`).
- `internal/agentpb/tsugi_agent.pb.go`, `..._grpc.pb.go` — new, generated
  (protoc-gen-go + -go-grpc), committed so the Docker build stays protoc-free.
- `internal/agent/server.go` — new. `Server` embeds
  `agentpb.UnimplementedTsugiAgentServer`; `var _ agentpb.TsugiAgentServer`.
- `Makefile` — `proto:` target (regenerate from the vendored .proto).
- `go.mod` / `go.sum` — added `google.golang.org/grpc` + `.../protobuf`.

# Previous Behavior

No gRPC; Tsugi was HTTP `/version` + `/healthz` + the `release` CLI only.

# New Behavior

None at runtime. The package compiles and exposes a server type whose RPCs return
`codes.Unimplemented`. Not registered in `cmd/tsugi`.

# Reason For Change

P5.1 contract slice — establish the typed cross-language seam before either side
implements it (read-before-write, Yagura PLAN §11/§13).

# Risks

- Proto drift: the contract is edited in Yagura, re-copied here. Keep them in sync
  (`make proto` after copying). One file, no buf/submodule yet.
- New build-host deps for regen: protoc + protoc-gen-go + protoc-gen-go-grpc. Not
  needed for `go build` (generated code is committed).

# Notes

Generated code committed deliberately so the existing protoc-free Dockerfile keeps
working. `go build ./...`, `go vet ./...`, `go test ./...` all green.
